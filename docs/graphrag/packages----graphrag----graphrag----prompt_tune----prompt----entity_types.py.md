
# `graphrag\packages\graphrag\graphrag\prompt_tune\prompt\entity_types.py` 详细设计文档

这是一个实体类型生成的微调提示词模板文件，定义了两种格式的提示词：一种返回逗号分隔的实体类型列表，另一种返回JSON格式的实体类型数组。主要用于大语言模型的实体提取任务中，指导模型识别文本中的实体类型。

## 整体流程

```mermaid
graph TD
    A[开始] --> B{选择提示词格式}
B -- 逗号分隔格式 --> C[使用ENTITY_TYPE_GENERATION_PROMPT]
B -- JSON格式 --> D[使用ENTITY_TYPE_GENERATION_JSON_PROMPT]
C --> E[替换{task}占位符]
D --> E
E --> F[替换{input_text}占位符]
F --> G[生成最终提示词]
G --> H[发送给语言模型]
H --> I{返回格式}
I -- 逗号分隔 --> J[解析字符串为列表]
I -- JSON格式 --> K[解析JSON为字典]
```

## 核心功能概述

该代码定义了用于从文本中提取实体类型的提示词模板，包含两种格式：一种输出逗号分隔的实体类型列表，另一种输出JSON格式的实体类型数组。这些提示词模板用于指导大语言模型根据用户任务从给定文本中识别相关的实体类型。

## 文件运行流程

该代码为纯配置性质的文件，不包含可执行逻辑，仅定义了两个字符串常量供其他模块调用。运行流程如下：
1. 模块被导入时，两个字符串常量被初始化
2. 调用方使用 `.format()` 方法填充 `{task}` 和 `{input_text}` 占位符
3. 填充后的提示词被传递给大语言模型API
4. 模型返回实体类型识别结果

## 关键组件信息

### 提示词模板结构

该组件包含完整的提示词设计，包含以下子部分：
- 任务目标说明：指导模型理解实体类型提取的目的
- 约束条件：避免通用实体类型、防止冗余重叠、强调质量优先
- 示例部分：提供三个示例供模型学习，但明确要求排除在最终答案外
- 真实数据部分：包含需要填充的任务和输入文本占位符

### 实体类型识别逻辑

该组件封装了实体类型提取的Prompt工程设计，包含：
- 任务相关性和上下文理解指导
- 输出格式规范（逗号分隔 vs JSON）
- 示例驱动的few-shot学习设计

## 技术债务与优化空间

1. **硬编码示例**：示例数据固定在模板中，无法动态调整，建议考虑将示例外置到配置或数据库
2. **缺乏验证机制**：没有对填充后的提示词进行有效性验证
3. **国际化支持不足**：提示词为英文，对于非英文场景可能需要重新设计
4. **重复代码**：两个提示词模板存在大量重复内容，可考虑提取公共部分
5. **占位符灵活性**：仅支持 `{task}` 和 `{input_text}` 两个占位符，扩展性有限

## 其他项目

### 设计目标与约束

- **目标**：为实体类型提取任务提供标准化的提示词模板
- **约束**：避免生成"other"、"unknown"等通用类型；防止冗余重叠；质量优先于数量

### 错误处理与异常设计

该代码不包含运行时错误处理逻辑，错误处理依赖调用方对模板填充过程的验证

### 数据流与状态机

数据流为单向：定义常量 → 格式填充 → 传递给LLM → 获取结果

### 外部依赖与接口契约

- 接口契约：调用方需提供 `task` 和 `input_text` 两个参数进行字符串格式化



## 问题及建议





### 已知问题

- **JSON示例格式不一致**：在`ENTITY_TYPE_JSON_PROMPT`的示例中，JSON数组元素没有使用引号（如`{"entity_types": [organization, person]}`），但实际有效的JSON应该是`{"entity_types": ["organization", "person"]}`，这可能导致LLM学习到错误的JSON格式
- **代码重复**：两个提示模板（普通版本和JSON版本）包含大量重复的指令和示例，只有输出格式不同，造成维护成本增加
- **硬编码的提示内容**：所有提示内容都是硬编码的字符串常量，没有提供配置化或参数化的方式来调整提示策略
- **缺少输入验证**：没有对模板变量`{task}`和`{input_text}`的占位符替换进行验证，如果替换不完整会导致提示失效
- **缺乏类型注解**：作为Python模块，没有使用类型提示（Type Hints），降低代码可读性和IDE支持
- **示例数据硬编码**：示例文本和任务描述是硬编码的，如果有新的使用场景需要更新示例，需要直接修改源代码
- **没有模块文档**：缺少模块级的docstring来说明这个文件的目的和使用方式

### 优化建议

- **修复JSON示例格式**：将示例中的数组元素添加引号，使其成为有效的JSON格式，如`{"entity_types": ["organization", "person"]}`
- **提取公共模板**：创建一个基础提示模板类或函数，将通用指令部分提取出来，通过参数控制输出格式，减少重复代码
- **引入配置管理**：使用配置文件或 dataclass 来管理提示的不同部分（指令、示例、格式要求），便于非开发人员调整
- **添加占位符验证**：在替换变量后验证所有占位符已被正确替换，或者使用f-string配合严格的格式化逻辑
- **添加类型注解**：为常量添加类型注解，如`ENTITY_TYPE_GENERATION_PROMPT: str = "..."`
- **外部化示例数据**：将示例数据抽取到独立的JSON或YAML配置文件中，便于维护和扩展
- **添加文档注释**：为模块添加docstring，说明其用途、依赖关系和使用注意事项
- **考虑单元测试**：添加针对模板格式化的单元测试，确保占位符替换的正确性

### 关键组件信息

### 提示模板变量

- **名称**: ENTITY_TYPE_GENERATION_PROMPT / ENTITY_TYPE_GENERATION_JSON_PROMPT
- **描述**: 两个预定义的提示模板，用于指导语言模型从输入文本中提取与任务相关的实体类型，避免冗余和通用类型

### 占位符机制

- **名称**: {task} / {input_text}
- **描述**: 提示模板中的动态占位符，分别用于插入用户任务描述和待分析的输入文本

### 设计目标与约束

- **设计目标**: 为实体类型提取任务提供标准化、结构化的提示词，确保模型输出质量一致
- **约束条件**: 禁止生成"other"、"unknown"等通用实体类型，要求避免冗余和重叠的实体类型

### 错误处理与异常设计

- **输入验证缺失**: 当前代码未对占位符替换进行验证，如果{task}或{input_text}未正确替换，可能导致模型输出格式错误
- **建议**: 在使用这些模板前应添加占位符完整性检查

### 数据流与状态机

- **数据流向**: 提示模板 → 字符串替换({task}, {input_text}) → 发送给LLM → 解析响应
- **状态**: 静态配置态（模板定义）→ 动态填充态（占位符替换）→ 请求发送态

### 外部依赖与接口契约

- **依赖**: 无代码级依赖，仅作为字符串资源被其他模块引用
- **接口契约**: 上游调用方需保证替换{task}和{input_text}占位符，接收模型返回的字符串或JSON并自行解析

### 潜在的技术债务或优化空间

1. **硬编码示例**: 示例文本被硬编码在提示模板中，如需更新示例需要修改源码，建议外部化配置
2. **缺乏输入验证**: 未验证占位符是否被正确替换，可能导致运行时错误
3. **重复结构**: 两个提示模板存在大量重复代码，可考虑抽象基类或模板组合
4. **国际化支持**: 当前仅支持英文提示词，如需支持其他语言需要重新设计
5. **模板复用性**: 提示模板专注于实体类型提取，通用性受限，考虑参数化扩展其他实体相关任务
