
# `graphrag\packages\graphrag\graphrag\query\context_builder\rate_prompt.py` 详细设计文档

这是一个用于评估信息相关性的prompt模板定义文件，包含一个名为RATE_QUERY的多行字符串模板，用于在全局搜索场景中让AI助手判断提供的描述信息与给定问题的相关性程度，评分范围为0-5分。

## 整体流程

```mermaid
graph TD
    A[开始] --> B[加载RATE_QUERY模板]
B --> C{需要评估信息相关性?}
C -- 是 --> D[填充{description}和{question}占位符]
D --> E[调用LLM进行相关性评分]
E --> F[解析JSON响应]
F --> G[提取reason和rating]
C -- 否 --> H[跳过相关性评估]
G --> I[结束]
H --> I
```

### 代码概述

该代码定义了一个用于评估信息相关性的全局搜索提示模板，通过预定义的Prompt结构引导AI助手对给定信息进行0-5分的相关性评分。

### 文件运行流程

该模块为静态配置模块，加载时直接定义RATE_QUERY字符串常量，供其他模块导入使用。运行时无实际执行逻辑，仅作为Prompt模板被引用。

### 全局变量详情

| 名称 | 类型 | 描述 |

| RATE_QUERY | str | 用于评估信息相关性的Prompt模板字符串，包含角色定义、目标说明、信息占位符、问题占位符和输出格式要求 |

### RATE_QUERY 模板结构

- **{description}**: 信息占位符，待评估的内容
- **{question}**: 问题占位符，用户提出的问题

### 关键组件信息

### RATE_QUERY 模板

Prompt模板组件，包含四个主要部分：角色定义（Role）、目标说明（Goal）、信息输入（Information）和问题输入（Question），以及JSON格式化的输出规范

### 潜在技术债务与优化空间

1. **硬编码模板问题**：Prompt模板直接硬编码在代码中，缺乏外部配置或国际化支持
2. **缺乏验证机制**：没有对模板参数（description、question）的校验
3. **JSON格式转义**：使用双花括号`{{}}`进行JSON转义，增加了模板复杂度
4. **字符串拼接方式**：未使用f-string或专门的模板引擎（如jinja2），灵活性受限

### 其它项目

**设计目标**：为全局搜索提供标准化的相关性评估Prompt，支持LLM进行信息过滤和优先级排序

**约束条件**：输出必须为JSON格式，包含reason和rating两个字段

**错误处理**：当前实现无错误处理机制，依赖调用方保证参数正确传递

**外部依赖**：无外部依赖，仅使用Python内置字符串类型



## 问题及建议



### 已知问题

-   **硬编码的Prompt模板**：`RATE_QUERY`作为魔法字符串直接硬编码在代码中，不利于国际化、多环境配置和后期维护
-   **缺少类型注解**：没有参数类型和返回值类型的注解，降低了代码的可读性和IDE支持
-   **缺乏文档注释**：缺少模块级和函数级的docstring，无法自动生成文档
-   **JSON格式定义不清晰**：在字符串中通过双花括号转义`{{`和`}}`来定义JSON结构，容易产生混淆和维护困难
-   **无输入验证机制**：没有对`description`和`question`参数进行有效性校验
-   **无错误处理**：调用方需要自行处理JSON解析失败等异常情况
-   **缺乏可测试性**：作为独立模块，没有提供测试接口或mock机制
-   **耦合度高**：prompt内容与业务逻辑紧耦合，无法灵活替换prompt策略

### 优化建议

-   **抽离配置**：将prompt模板移至独立的配置文件（如YAML/JSON）或环境变量，支持动态加载
-   **添加类型注解**：为函数参数和返回值添加`typing`模块的类型定义
-   **结构化输入输出**：使用`dataclass`或`pydantic`定义请求和响应结构，提供自动验证
-   **添加文档字符串**：编写清晰的模块和函数文档，说明用途、参数和返回值
-   **封装为函数/类**：将prompt模板封装为可配置的函数，支持参数化定制
-   **考虑模板引擎**：对于复杂的prompt，可考虑使用Jinja2等模板引擎提升可维护性
-   **添加单元测试**：为解析逻辑、默认值处理等编写测试用例
-   **日志记录**：添加适当的日志输出，便于调试和问题追踪

## 其它





### 设计目标与约束

该代码模块的设计目标是提供一个标准化的信息相关性评估提示模板，用于全局搜索中动态选择社区时判断信息片段与查询问题的相关程度。约束条件包括：必须返回JSON格式的评分结果，评分范围限定为0-5的整数，且提示词需要兼顾推理过程的说明和评分的准确性。

### 错误处理与异常设计

由于该代码仅为一个字符串常量，不涉及运行时错误处理逻辑。若在使用该提示模板的调用方发生错误，通常由调用方负责捕获和处理可能出现的JSON解析异常（如返回格式不符合预期）、网络超时异常（如LLM调用超时）以及提示模板变量未正确填充导致的占位符遗漏问题。

### 数据流与状态机

该模块不涉及复杂的状态机或数据流处理。其数据流为：调用方提供`{description}`和`{question}`两个变量 → 填充到RATE_QUERY模板中 → 传递给LLM → LLM返回JSON格式的评分结果 → 调用方解析结果。整个过程为单向数据流，无状态保持需求。

### 外部依赖与接口契约

该模块无直接外部依赖，仅作为字符串模板被其他模块引用。使用该模板的调用方需要满足以下接口契约：
- 输入：提供`description`（待评估信息）和`question`（目标问题）两个字符串参数
- 输出：期望获得包含`reason`（字符串类型）和`rating`（0-5整数）两个字段的JSON对象

### 性能考虑

该模块为纯字符串常量，不产生运行时性能开销。在实际使用中，性能瓶颈主要取决于调用LLM进行推理的延迟和网络通信时间。建议对LLM调用设置合理的超时时间，并在高并发场景下考虑实现缓存机制以避免重复评估相同信息。

### 安全考虑

该代码本身不涉及敏感数据处理，但使用场景中需要注意：
- 传入的`description`可能包含敏感信息，需确保LLM服务提供商的数据安全政策
- 提示词中未包含显式的安全过滤指令，建议在实际应用中根据需求添加内容安全相关的约束条件

### 配置说明

该模块无配置参数。但使用该模板时可配置以下参数：
- LLM模型选择（影响评分一致性）
- LLM温度参数（影响评分稳定性）
- 超时时间设置
- 重试策略配置
