
# `graphrag\packages\graphrag\graphrag\prompt_tune\template\entity_summarization.py` 详细设计文档

这是一个用于实体摘要任务的微调提示模板，通过定义角色人格、语言要求和输入输出格式，指导AI模型将多个关于同一实体的描述整合为一个连贯、简洁的第三人称摘要，支持矛盾信息的整合和上下文丰富化。

## 整体流程

```mermaid
graph TD
    A[开始] --> B[定义提示模板字符串]
    B --> C[包含角色人格占位符 {persona}]
    C --> D[包含语言占位符 {language}]
D --> E[定义输入变量: entity_name, description_list]
E --> F[定义输出格式要求: 第三人称、包含实体名]
F --> G[定义处理逻辑: 整合描述、解决矛盾、丰富信息]
G --> H[结束: 生成静态提示模板供外部调用]
```

## 关键组件





### 实体摘要提示词模板 (ENTITY_SUMMARIZATION_PROMPT)

这是一个全局字符串常量，定义了用于实体摘要任务的提示词模板，包含了角色设定(persona)、语言参数(language)、实体名称(entity_name)和描述列表(description_list)等占位符，用于动态生成摘要。

### 角色设定占位符 ({persona})

提示词模板中的动态占位符，允许用户定义AI助手的角色身份和专业背景，以指导摘要生成的语气和专业深度。

### 语言参数占位符 ({language})

动态占位符，指定摘要输出的语言，确保摘要内容以指定语言呈现。

### 实体名称占位符 ({entity_name})

动态占位符，用于填充需要摘要的实体名称，确保输出包含完整的上下文信息。

### 描述列表占位符 ({description_list})

动态占位符，接收一个或多个与实体相关的描述列表，用于聚合和整合信息。

### 矛盾处理逻辑

提示词中的条件逻辑指令，要求模型识别并解决输入描述中的矛盾信息，生成连贯统一的摘要。

### 第三人称写作指令

明确要求摘要使用第三人称视角撰写，确保生成内容的客观性和专业性。

### 信息丰富化指令

强调利用附近文本信息丰富摘要内容，这是实体摘要的重要优化策略。

### 空输入处理逻辑

当输入为空或无法生成有效答案时，仅输出文本中提供的有限信息，避免幻觉。



## 问题及建议




### 已知问题

-   **占位符语法不一致**：代码中使用了两种不同的占位符语法，单大括号 `{persona}` 和 `{language}` 与双大括号 `{{entity_name}}` 和 `{{description_list}}` 混用，这种不一致可能导致格式化时的混淆或错误
-   **缺乏输入验证**：模板没有对 `entity_name` 和 `description_list` 为空或无效值的情况进行处理，可能导致生成无意义或错误的摘要
-   **模板注入风险**：占位符直接替换而没有进行输入清理或转义，如果用户输入被直接代入，可能存在安全风险
-   **缺少错误处理**：当必需的占位符未被提供时，没有回退机制或默认值处理
-   **提示词分隔符安全隐患**："#######" 分隔符未对用户输入进行过滤，可能被恶意用户利用进行提示注入攻击
-   **无文档说明**：常量缺少 docstring 或注释来说明其用途、使用方法和格式要求
-   **紧耦合设计**：模板与特定的变量名紧密绑定，降低了可复用性和可测试性

### 优化建议

-   统一占位符语法，建议全部使用双大括号 `{{variable}}` 以避免与 Python f-string 格式化冲突，或明确区分两种语义的占位符
-   添加输入验证逻辑，检查必需字段是否存在且有效，提供明确的错误信息
-   对用户输入进行转义或清理，处理特殊字符以防止提示注入
-   添加可选的默认占位符处理机制，当变量缺失时使用空字符串或合理的默认值
-   增加 docstring 文档，说明模板的用途、参数说明和使用示例
-   考虑将提示模板封装为类或使用配置管理，提供类型提示和更结构化的管理方式
-   将分隔符改为更独特且不易被用户输入混淆的形式，或对用户输入进行边界处理


## 其它




### 1. 一段话描述

该代码定义了一个用于实体摘要（Entity Summarization）的提示词模板，通过大语言模型将一个或多个实体的多个描述信息整合为一个连贯、全面、第三人称视角的摘要文本，支持多语言输出，并能够处理描述中的矛盾信息。

### 2. 文件的整体运行流程

该文件为配置型代码，不涉及运行时执行流程。其使用流程如下：
1. 外部系统加载该模块
2. 通过字符串格式化方法（str.format() 或 f-string）填充模板中的占位符：`{persona}`（角色设定）、`{language}`（目标语言）、`{{entity_name}}`（实体名称）、`{{description_list}}`（描述列表）
3. 将填充后的完整提示词发送给大语言模型
4. 模型根据提示词生成实体摘要并返回

### 6. 潜在的技术债务或优化空间

1. **缺乏参数验证机制**：模板变量填充时没有进行格式校验，可能导致注入无效内容
2. **国际化支持不完善**：仅通过 `{language}` 占位符支持多语言，但缺乏语言代码规范和默认语言设定
3. **模板与业务逻辑耦合**：提示词内容硬编码在代码中，不利于非技术人员调整优化
4. **缺少版本管理**：提示词迭代演进时缺乏版本控制和变更记录
5. **错误处理缺失**：当描述列表为空或格式异常时，没有明确的错误处理逻辑
6. **可测试性不足**：由于是纯字符串定义，无法直接进行单元测试验证输出质量

### 7. 其它项目

#### 设计目标与约束

- **目标**: 为大语言模型提供标准化的实体摘要生成指令，输出连贯、简洁、第三人称视角的摘要文本
- **约束**: 
  - 输入必须是实体名称和描述列表
  - 输出语言由 `{language}` 参数指定
  - 强制使用第三人称书写
  - 需解决描述中的矛盾信息

#### 错误处理与异常设计

- **当前状态**: 无显式错误处理机制
- **建议改进**:
  - 验证 `entity_name` 不为空
  - 验证 `description_list` 格式正确（非空列表）
  - 对 `language` 参数进行合法性校验（如支持的语言列表）
  - 处理 `persona` 为空时的默认值设定

#### 数据流与状态机

- **数据输入**: 
  - `persona`: 字符串，模型角色设定
  - `language`: 字符串，目标语言（如 "en"、"zh"）
  - `entity_name`: 字符串，待摘要的实体名称
  - `description_list`: 列表，实体相关描述集合
- **数据处理**: 模板字符串格式化，变量替换
- **数据输出**: 填充完整的提示词字符串，发送给大语言模型

#### 外部依赖与接口契约

- **外部依赖**: 
  - Python 字符串格式化模块（str.format() 或字符串插值）
  - 调用大语言模型的外部系统（依赖方）
- **接口契约**:
  - 模块导出 `ENTITY_SUMMARIZATION_PROMPT` 全局变量
  - 调用方需使用 `.format()` 方法或 f-string 填充四个占位符
  - 期望 `entity_name` 和 `description_list` 使用双花括号 `{{}}` 转义，以在格式化后保持变量名不变

#### 安全性考虑

- **提示注入风险**: `persona` 和描述列表内容未经过滤，可能存在提示注入攻击风险
- **建议**: 对用户输入的内容进行安全过滤和长度限制

#### 可维护性与扩展性

- **当前限制**: 提示词修改需要修改源代码
- **建议改进**: 
  - 考虑将提示词模板外置到配置文件或数据库
  - 支持通过环境变量或配置系统动态加载不同版本的提示词
  - 添加提示词版本号和变更日志
